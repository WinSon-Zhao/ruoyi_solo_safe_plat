## 1. 项目概述

### 1.1 建设目标

建设一套符合国标要求的安全设备日志采集分析系统（态势感知平台），实现：

1. **统一接入**内网各类安全设备，页面配置对接新设备
2. **标准化解析**异构日志，字段字典映射
3. **安全分析**（威胁情报、漏洞库、资产匹配、ATT&CK攻击链）
4. **告警聚合**为安全事件，分级分类
5. **运维处置**（封堵IP、隔离主机、联动防火墙）
6. **数据输出与上报**（国标格式、监管对接）
7. **溯源检索**（ES 全文检索、攻击链还原）

### 1.2 适用国标清单

| 国标 | 名称 | 核心要求 | 系统对应 |
|------|------|----------|----------|

## 2. 技术选型

### 2.1 技术栈

| 层级 | 技术 | 版本 | 选型理由 |
|------|------|------|----------|
| **后端框架** | Solon | 3.2+ | 国产框架，启动快、内存低，注解兼容 Spring |
| **基础框架** | ruoyi-solon（单体版） | 最新 | 提供用户/角色/菜单/字典等后台管理基础能力，基于 Solon + Vue 前后端分离 |
| **ORM** | MyBatis-Flex | 1.7+ | 轻量高效，Solon 生态友好 |
| **消息队列** | Kafka | 3.7+ | 设备接入与系统内部共用，Topic 隔离 |
| **数据库** | MySQL | 8.0+ | 配置数据、聚合告警、规则管理 |
| **搜索引擎** | EasySearch | 1.0+ | ES 兼容，国产化替代，日志存储与检索 |
| **缓存** | Redis | 6.0+ | 字典缓存、会话、聚合窗口、限流 |
| **脚本引擎** | GraalJS | 23.0+ | 动态接口，修改即生效，无需重启 |
| **前端** | 原生 HTML/JS/CSS | - | 无构建步骤，AI 可直接生成，热加载 |
| **构建工具** | Maven | 3.8+ | 多模块项目管理 |

### 2.2 选型决策记录

| 决策点 | 选项 | 选择 | 理由 |
|--------|------|------|------|
| 后端框架 | Spring Boot vs Solon | **Solon** | 国产化要求，学习成本低 |
| 前端框架 | Vue 3 vs 原生 HTML | **原生 HTML** | AI 友好，无构建步骤，热加载 |
| 搜索引擎 | Elasticsearch vs EasySearch | **EasySearch** | 国产化，ES API 兼容 |
| 规则引擎 | Drools vs GraalJS | **GraalJS** | 轻量、配置化、AI 易生成 |
| 数据处理 | Flink vs Kafka 窗口 | **Kafka 窗口** | MVP 够用，降低复杂度 |
| 工作流 | Flowable vs 状态机 | **状态机** | 轻量，告警处置流程简单 |

### 3.2 模块清单与职责

| 模块 | 说明 | 阶段 |
|------|------|------|
| **common** | DTO、常量、工具类、异常 | 阶段0 |
| **api** | 模块间服务接口定义（数据总线），单体模式下类调用，微服务模式下 RPC 调用 | 阶段0 |
| **config** | 动态配置管理、脚本引擎、热加载、字典管理 | 阶段0 |
| **gateway** | 路由、JWT认证、限流、静态资源、动态脚本路由 | 阶段0 |
| **device** | 设备CRUD、日志接收（Syslog/HTTP/Kafka）、模拟设备 | 阶段1 |
| **parse** | 解析规则管理、JSON/正则/XML/Grok解析、字段映射、字典转换 | 阶段1 |
| **analysis** | 规则引擎、黑名单、威胁情报、漏洞匹配、ATT&CK映射 | 阶段1 |
| **alert** | 告警聚合、告警CRUD、统计展示、事件聚合 | 阶段1 |
| **disposal** | 处置配置、IP封堵、主机隔离、处置记录 | 阶段2 |
| **report** | 上报配置、数据格式转换、监管对接 | 阶段2 |
| **ops** | 运维验证平台（连接管理、MySQL/Redis/Kafka/ES运维、API测试、脚本管理） | 阶段0 |

> **架构演进策略**：
> - **阶段0-3（MVP）**：基于 ruoyi-solon 单体版改造，所有模块打包为一个 JAR（security-platform-admin），统一端口 :8080 启动
> - **阶段4（生产化）**：迁移到 ruoyi-cloud-solon 微服务版，核心业务代码零修改
> - **api 模块**：定义模块间服务接口，单体模式下通过 Solon 注入直接类调用，微服务模式下自动切换为 RPC 调用
> - **异步流水线**：日志采集→解析→分析→告警 链路始终通过 Kafka 异步通信（高吞吐、解耦、削峰）

### 3.3 模块依赖关系

```
                    ┌──────────┐
                    │ gateway  │
                    └────┬─────┘
                         │
       ┌─────────┬───────┼───────┬─────────┐
       ↓         ↓       ↓       ↓         ↓
  ┌────────┐ ┌────────┐ ┌───┐ ┌────────┐ ┌────────┐
  │ device │ │ parse  │ │...│ │ alert  │ │ report │
  └───┬────┘ └───┬────┘ └───┘ └───┬────┘ └───┬────┘
      │           │               │           │
      └───────────┴───────┬───────┴───────────┘
                         │
              ┌──────────┼──────────┐
              ↓          ↓          ↓
         ┌────────┐ ┌────────┐ ┌────────┐
         │  api   │ │ common │ │ config │
         └────────┘ └────────┘ └────────┘
```

**依赖规则**：
- 所有业务模块依赖 common 和 api，common 不依赖任何业务模块
- **api 模块**（数据总线）：定义模块间服务接口，各模块只依赖接口不依赖实现
  - 单体模式：Solon 容器自动注入实现类，等效于类调用
  - 微服务模式：自动切换为 Solon Cloud RPC 调用，业务代码无需修改
- **异步流水线**（日志→解析→分析→告警）：始终通过 Kafka Topic 通信
- **同步查询**（告警查设备、事件查资产等）：通过 api 模块接口调用
- 禁止循环依赖，禁止跨模块直接 `@Inject` 实现类

## 4. 数据流设计

### 4.1 核心数据流（日志采集→告警展示）

```
安全设备 ──→ [Syslog/HTTP/Kafka] ──→ device模块 ──→ Kafka:security-events-raw
                                                              │
                                                        parse模块（消费）
                                                              │
                                                     字段映射 + 字典转换
                                                              │
                                                    Kafka:security-events-parsed
                                                              │
                                                      analysis模块（消费）
                                                              │
                                                  规则引擎（黑名单/威胁情报/漏洞/ATT&CK）
                                                              │
                                                    Kafka:security-events-alert
                                                              │
                                                       alert模块（消费）
                                                         ┌────┴────┐
                                                         ↓         ↓
                                                   EasySearch    MySQL
                                                   (原始告警)   (聚合告警)
                                                         │         │
                                                         ↓         ↓
                                                      前端展示页面
```

> **通信模式说明**：
> - 上面的日志流水线使用 **Kafka 异步通信**（高吞吐、解耦、削峰），单体/微服务模式下行为一致
> - 前端页面查询告警详情时，alert 模块通过 **api 模块** 同步调用 device 模块获取设备信息（单体→类调用，微服务→RPC）

### 4.2 处置数据流

```
告警详情页 ──→ [一键封堵] ──→ disposal模块 ──→ 防火墙API ──→ 执行结果
                                              │
                                              ↓
                                         MySQL:处置记录
```

### 4.3 上报数据流

```
analysis模块 ──→ Kafka:security-events-report ──→ report模块 ──→ 监管平台
```

### 4.4 Kafka Topic 规划

| Topic | 生产者 | 消费者 | 数据 | 保留 |
|-------|--------|--------|------|------|
| `device.{code}.raw` | 安全设备 | device | 设备原始推送 | 24h |
| `security-events-raw` | device | parse | 标准化原始事件 | 24h |
| `security-events-parsed` | parse | analysis | 解析后统一事件 | 72h |
| `security-events-alert` | analysis | alert | 告警事件 | 72h |
| `security-events-report` | analysis | report | 上报数据 | 72h |
| `security-config-change` | config | all | 配置变更通知 | 24h |
| `security-events-dlq` | parse | 人工 | 解析失败死信 | 7d |

## 5. 统一数据模型

### 5.1 设计原则

- 基于 ECS（Elastic Common Schema）扩展
- **基础字段 + 扩展字段**模式，category 区分类型
- 字段命名 snake_case
- 新增类型只加扩展字段，不改基础结构

### 5.2 SecurityEvent（基础模型，所有类型共享）

| 字段 | 类型 | 说明 |
|------|------|------|
| event_id | String | 唯一ID（UUID） |
| timestamp | Date | 事件发生时间 |
| received_time | Date | 系统接收时间 |
| category | String | alert / audit / flow |
| severity | String | critical / high / medium / low / info |
| device_id | String | 设备ID |
| device_vendor | String | 设备厂商 |
| device_type | String | firewall / ids / waf / edr / ... |
| source_ip | String | 源IP |
| source_port | Integer | 源端口 |
| destination_ip | String | 目的IP |
| destination_port | Integer | 目的端口 |
| protocol | String | TCP / UDP / HTTP / ... |
| action | String | allow / deny / alert / block |
| raw_message | Text | 原始日志 |
| tags | String[] | 标签 |

### 5.3 扩展模型

| 模型 | category | 阶段 | 扩展字段 |
|------|----------|------|----------|
| AlertEvent | alert | MVP | alert_name, alert_type, rule_id, attack_type, threat_intel_hit, vuln_id, tactic_id, technique_id, confidence |
| AuditEvent | audit | 二期 | audit_type, user_name, operation, object_name, http_method, http_url, bytes_in, bytes_out |
| FlowEvent | flow | 三期 | flow_direction, bytes, packets, duration, dns_domain, app_protocol |

### 5.4 安全事件模型（告警聚合为事件）

| 字段 | 类型 | 说明 |
|------|------|------|
| incident_id | String | 事件编号 |
| incident_level | String | 一般 / 较大 / 重大 / 特别重大（GB/T 37027） |
| attack_chain | JSON | ATT&CK 战术+技术+时间线 |
| related_alerts | JSON | 关联告警ID列表 |
| source_ips | JSON | 所有攻击源IP |
| target_assets | JSON | 受影响资产 |

## 7. 关键技术方案

### 7.1 解析引擎（策略模式）

```
LogParser 接口
├── JsonParser    （JSONPath 提取）
├── RegexParser   （正则命名组提取）
├── XmlParser     （XPath 提取，二期）
└── GrokParser    （Grok 模式，二期）
```

- 通过 ParseRule.rule_type 自动选择 Parser
- 解析失败写入死信 Topic
- 新增格式只需实现 ParserPlugin 接口

### 7.2 安全规则引擎（责任链模式）

```
SecurityRule 接口
├── BlacklistRulePlugin     （IP/域名黑名单）
├── ThreatIntelRulePlugin   （威胁情报匹配）
├── VulnRulePlugin          （漏洞匹配，MVP）
├── AssetRulePlugin         （资产关联，二期）
└── CorrelationRulePlugin   （关联分析，三期）
```

- 规则按 priority 排序，支持短路
- 规则配置存 MySQL，缓存到 Redis，支持热更新
- 新增规则类型只需实现 RulePlugin 接口

### 7.3 告警聚合（时间窗口）

- MVP：固定窗口（5分钟），维度 = source_ip + alert_type + device_id
- Redis Key 过期实现窗口滑动
- 二期：滑动窗口 + 多维度聚合

### 7.4 动态脚本（GraalJS）

- scripts/ 目录下放 .js 文件，修改即生效
- 脚本可定义 API 路由，由 gateway 动态加载
- 注入 services 对象（db/redis/kafka/log）
- static/pages/ 下放 HTML，浏览器刷新即更新

### 7.5 处置联动（插件模式）

```
DisposalPlugin 接口
├── FirewallBlockAction     （防火墙IP封堵）
├── EDRIsolateAction        （EDR主机隔离）
├── WAFUrlFilterAction      （WAF URL过滤）
└── SwitchPortControlAction （交换机端口管控）
```

- 处置配置页面可配（对接方式、认证、参数模板）
- 执行结果回调，记录到 MySQL

### 7.6 简单处理跳过

- 设备配置 `skip_analysis = true`
- 解析后直接写 ES，不经过 analysis 模块
- 通过 Kafka Topic 路由实现

## 9. AI 开发策略

### 9.1 代码分层与 AI 生成率

| 层级 | 内容 | AI 生成率 | 人工介入 |
|------|------|-----------|----------|
| L1 配置 | YAML/JSON/SQL/HTML/CSS | 95% | 仅审核 |
| L2 模板 | Entity/Mapper/Service/Controller CRUD | 90% | 仅审核 |
| L3 业务 | 解析引擎/规则匹配/告警聚合 | 70% | Review 核心算法 |
| L4 复杂 | 攻击链关联/事件分级 | 40% | AI 辅助+人工主导 |
| L5 核心 | 消息封装/脚本引擎/安全过滤 | 20% | 人工设计+AI 实现 |

### 9.2 大模型介入节点

| 节点 | 方式 | 产出 |
|------|------|------|
| 需求分析 | 人工描述→AI 补充 | 需求文档 |
| 详细设计 | AI 生成→人工审核 | 设计文档 |
| 代码开发 | AI 生成→人工 Review | 可运行代码 |
| 测试验证 | AI 生成数据/用例→人工执行 | 测试报告 |
| 运维优化 | AI 分析日志→建议方案 | 优化建议 |

### 9.3 架构演进策略（单体→微服务平滑迁移）

**演进路线**：

```
阶段0-3（MVP）                          阶段4（生产化）
ruoyi-solon 单体版          ──→         ruoyi-cloud-solon 微服务版
单 JAR :8080                            多服务 + Nacos + Gateway
```

**核心设计：api 数据总线模块**

```
┌─────────────────────────────────────────────────────────────────┐
│                        api 模块（数据总线）                       │
│                                                                 │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐ │
│  │ DeviceApi       │  │ AlertApi        │  │ AssetApi        │ │
│  │ (设备服务接口)   │  │ (告警服务接口)   │  │ (资产服务接口)   │ │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘ │
│  ┌─────────────────┐  ┌─────────────────┐                      │
│  │ ParseApi        │  │ DisposalApi     │  ...                 │
│  │ (解析服务接口)   │  │ (处置服务接口)   │                      │
│  └─────────────────┘  └─────────────────┘                      │
│                                                                 │
│  各模块只依赖 api 中的接口，不依赖实现类                           │
└─────────────────────────────────────────────────────────────────┘
```

**两种模式下的调用方式**：

| 调用场景 | 单体模式（阶段0-3） | 微服务模式（阶段4） |
|----------|-------------------|-------------------|
| alert 查询设备信息 | Solon 容器注入 DeviceApiImpl，直接类调用 | Solon Cloud 自动代理为 RPC 调用 |
| event 查询告警列表 | Solon 容器注入 AlertApiImpl，直接类调用 | Solon Cloud 自动代理为 RPC 调用 |
| 日志→解析→分析→告警 | Kafka 异步（与微服务模式完全一致） | Kafka 异步 |

**开发阶段必须遵守的规则**：

| 规则 | 说明 | 违反后果 |
|------|------|----------|
| 跨模块调用只依赖 api 接口 | 禁止 `@Inject DeviceServiceImpl`，只能 `@Inject DeviceApi` | 违反则微服务迁移时需改代码 |
| 异步流水线用 Kafka | 日志采集→解析→分析→告警 始终走 Kafka Topic | 违反则单体→微服务行为不一致 |
| Service 无状态 | 不依赖 JVM 级缓存/变量，用 Redis 代替 | 违反则多实例部署行为不一致 |
| 配置外置 | 连接参数放 application.yml，不硬编码 | 违反则迁移时需改代码 |
| 禁止循环依赖 | 模块间依赖方向：业务模块 → api/common | 违反则编译失败 |
